## 0. 철학 & 원칙

### 0.1 "감" 금지, "지표" 기반

게임 밸런스 조정에서 가장 흔한 실패는 **직관 기반 조정**:
- "요즘 Lv5 유저가 많아진 것 같다" → 근거 없이 상향 → 유저 이탈
- "다들 어려워한다는 피드백이 있다" → 일부 목소리 큰 유저에 반응 → 대다수 불만족

**글러브는 4가지 객관 트리거**(§2)가 발동해야만 조정 검토를 시작한다.

### 0.2 "한 번 바꾸면 되돌리기 어렵다"

- 레벨 상향 후 유저 이탈 발생 시 "원래대로" 해도 이미 떠난 유저는 돌아오지 않음
- 수수료 조정은 크리에이터 신뢰에 직접 영향 → 연 1회 이하
- **모든 조정은 "신중한 결정"이 기본**

### 0.3 유튜브 사례 참고

유튜브 YPP(Partner Program) 조정 이력:
- 2012년: 구독자 제한 없음
- 2017년: 1만 조회수 요구 (소규모 저품질 차단)
- 2018년: 1천 구독 + 4천 시간 (현재 기준)
- 2023년: 500 구독 + 3천 시간 (완화)

→ **11년간 4번 조정**. 매번 **공개 데이터 분석 + 수개월 예고**.

### 0.4 개발 수칙 (CLAUDE.md 준수)

- Strict Focus: 조정 외 코드 변경 금지
- Surgical Edit: 상수만 수정, 로직은 그대로
- 선보고 후실행: 조정안 AS-IS/TO-BE → 유저 공지 → 실행

## 2. 4대 판단 트리거

> 조정 검토를 시작하려면 아래 4가지 중 **2개 이상** 해당되어야 함.
>
> **관찰용 대시보드가 이 데이터를 실시간 집계** (§6 참조, 실제 UI는 `ADMIN.md`).

### 2.1 트리거 1: Lv 분포 이상

**정상 분포** (유튜브·트위치 참고):

| 레벨 | 정상 비율 | 이상 신호 |
|:----:|:--------:|:---------:|
| Lv1~2 | 50~70% | < 30% (초기 유지율 낮음) |
| Lv3~4 | 20~30% | > 50% (너무 쉬움) |
| Lv5~6 | 5~10% | > 20% (수익 게이트 느슨) |
| Lv7~8 | 1~3% | > 5% |
| Lv9~10 | < 1% | > 2% |

**조치**:
- Lv5 이상 > 15% → 🔴 상향 조정 검토
- Lv5 이상 < 1% → 🟢 하향 조정 검토

### 2.2 트리거 2: 수익 창출 전환율

**체크 지표**: "가입 → 첫 수익(가판대·단골장부·광고) 발생까지 걸린 시간"

| 기간 | 평가 | 조치 |
|:----:|:----:|------|
| 1개월 이내 | 🔴 너무 쉬움 | Lv3, Lv5 상향 검토 |
| 3~6개월 | 🟢 적정 (유튜브 YPP 평균) | 유지 |
| 1년+ | 🔴 너무 어려움 | 하향 검토 |

**추가 지표**: "가입 후 1년 내 최소 1건 이상 수익 발생한 유저 비율"
- 5~15% = 적정
- < 5% = 너무 어려움
- > 20% = 너무 쉬움 (품질 저하 우려)

### 2.3 트리거 3: 어뷰저 비율

**체크 지표**: `detectCircularThanksball`·`auditReputationAnomalies` 등 탐지에 걸린 유저 비율 (`ANTI_ABUSE.md` 참조)

| 비율 | 평가 | 조치 |
|:----:|:----:|------|
| > 1% | 🔴 현재 기준이 어뷰저에게 유리 | Lv/평판 임계값 상향 |
| 0.1~1% | 🟡 모니터링 | 즉시 조치 불필요 |
| < 0.1% | 🟢 적정 | 유지 |

**주의**: 어뷰저 비율이 높다고 **무조건 임계값 상향**은 오답. 먼저 **탐지 로직 강화**가 정답. 임계값 상향은 최후 수단.

### 2.4 트리거 4: 정성 피드백

**소스**:
- 고객센터 문의
- 커뮤니티 내 글/댓글 분석
- 앱스토어 리뷰
- 직접 인터뷰

**평가 기준**:
- 5% 이상 유저가 "너무 쉽다" → 상향 검토
- 5% 이상 유저가 "너무 어렵다" → 하향 검토
- 피드백 수가 통계적 유의미해야 함 (최소 50건+)

**주의**:
- 소수 "목소리 큰" 유저에 과민 반응 금지
- 피드백은 보조 지표, 정량 지표 우선

### 2.5 종합 판정 규칙

**조정 검토 시작 조건**:
- 트리거 1~3 중 2개 이상 🔴 + 트리거 4가 🟡 이상

**조정 거부 조건**:
- 트리거 1~3 모두 🟢 또는 🟡
- 피드백만 🔴 (다수결 조작 우려)

## 3. 3단계 타임라인 (Phase A → B → C)

### 3.1 전체 로드맵

```
[Phase A] 현재 — 2026년 베타
     │ 느슨한 기준 유지
     │ 지표 수집만
     │ grandfatheredLevel 코드 미구현
     │
     ↓ (베타 종료 + PG 연동 직전)
     
[Phase B] 베타 종료 공지 + 전면 재조정
     │ 2~3개월 전 예고
     │ Phase A 데이터 기반 완전 재설계
     │ 리셋 분위기 (테스트였음 명시)
     │
     ↓ (정식 출시)
     
[Phase C] 정식 출시 이후
     │ A+B+E 전략 가동
     │ grandfatheredLevel 구현
     │ 조정 시 기득권 보장 필수
     │ 연 1~2회 신중한 조정
```

### 3.2 Phase A: 현재 베타 (~ Lv10 유저 100명)

**상태**:
- 🔵 **지금 여기**
- 2026년 4월 현재, 테스트 계정 14개 외 소수 알파 유저

**행동 원칙**:
- **조정 금지** (데이터 부족)
- 상수 분리만 진행 (§8)
- 관찰용 대시보드 구축 (§6 요구사항, 실제는 `ADMIN.md`)

**종료 조건** (다음 Phase로 이동):
- Lv10 유저 100명 돌파 **OR** 정식 서비스 출시 6개월 전

### 3.3 Phase B: 베타 종료 + 전면 재조정

## 4. 마이그레이션 전략 (A+B+E 조합)

> **적용 시점**: Phase C(정식 출시 이후)부터
>
> Phase B(베타 종료)에서는 **리셋 성격**이므로 grandfather 불필요

### 4.1 3가지 접근 조합

#### A. Grandfather Clause (기득권 보장)

**상향 조정 시 필수**

**구현**:
```typescript
// types.ts UserData 확장
export interface UserData {
  exp: number;
  grandfatheredLevel?: number;  // 🆕 조정 전 최고 달성 레벨
  grandfatheredAt?: FirestoreTimestamp;  // 🆕 스냅샷 시점
}

// utils.ts calculateLevel 수정
export const calculateLevel = (userData: UserData | { exp: number }): number => {
  const exp = 'exp' in userData ? (userData.exp || 0) : 0;
  let computedLevel = 1;
  for (let i = LEVEL_REQUIREMENTS.length - 1; i >= 0; i--) {
    if (exp >= LEVEL_REQUIREMENTS[i]) {
      computedLevel = i + 1;
      break;
    }
  }
  // 🆕 기득권 보장
  if ('grandfatheredLevel' in userData && userData.grandfatheredLevel) {
    return Math.max(computedLevel, userData.grandfatheredLevel);
  }
  return computedLevel;
};
```

#### B. Announcement Period (3개월 예고)

**모든 조정의 기본값**

**공지 템플릿**:
```
📢 경계값 조정 예정 안내

2026-XX-XX부터 다음 기준이 조정됩니다.

[변경 내용]
- Lv5 EXP 요구: 500 → 700

[기존 유저 보호]
- 이미 Lv5+ 달성하신 분들은 영구 유지
- grandfatheredLevel: 5 자동 기록

[이유]
- Lv5 이상 유저 비율이 18%에 도달하여 상위 티어의 희소성 확보 필요

상세 FAQ: [링크]
```

#### E. 하향은 즉시, 상향만 보호

**철학**:
- **하향** (유리): 모든 유저 이득 → 예고 불필요, 즉시 적용
- **상향** (불리): 기존 유저 보호 필수 (A+B 동시 적용)

### 4.2 조합 매트릭스

| 조정 성격 | 전략 | 예시 |
|-----------|------|------|
| 레벨 상향 | B(예고) + A(기득권) | Lv5: 500 → 700 |
| 레벨 하향 | E(즉시) | Lv5: 500 → 400 |
| 평판 상향 | B + A | 확고: 3000 → 5000 |
| 평판 하향 | E | 확고: 3000 → 2500 |
| 마패 경계 상향 | B + A | 금마패: 2.0 → 2.5 Score |
| 마패 경계 하향 | E + 알림 | 금마패: 2.0 → 1.5 Score |
| 수수료 증가 | B (3개월 예고) | 시장 수수료 25% → 30% |
| 수수료 감소 | E | 25% → 20% |

### 4.3 예외 케이스: 수수료는 "거래 스냅샷" 방식

**기존 거래는 그대로 유지**:
- `market_purchases.feeRate` 필드가 **거래 시점 수수료율 저장**
- 신규 거래부터만 새 수수료율 적용
- 정산 시 각 거래의 저장된 feeRate 사용

**→ 수수료 조정은 마이그레이션 불필요** (설계상 이미 스냅샷 패턴)

### 4.4 대폭 조정 (>30% 변화)

**추가 접근 C (점진적 적용, Gradual Ramp)**:
- 예: Lv5 500 → 1000 (100% 증가)
- 6개월에 걸쳐 월 15%씩 점진 상향
- 충격 분산, "한 번에 강등" 사태 방지

**구현 방식**:
```typescript
// constants.ts
export const LEVEL_REQUIREMENTS_SCHEDULE = {
  '2026-10-01': [0, 30, 100, 250, 500, 1000, ...],     // 현재
  '2026-11-01': [0, 30, 100, 250, 575, 1100, ...],     // 1차
  '2026-12-01': [0, 30, 100, 250, 650, 1200, ...],     // 2차
  // ...
  '2027-04-01': [0, 30, 100, 250, 1000, 2000, ...],    // 최종
};

// 현재 적용 테이블 = 오늘 날짜 이전 가장 최근 스케줄
```

**단점**:
- 복잡도 높음 (매월 배포, 유저 혼란)
- 구현 비용 큼

**권장 사용**:
- 정말 필요한 경우만 (50%+ 변화 시)
- A+B+E 조합만으로 부족한 극단적 상황

### 4.5 칭호·마패의 마이그레이션 특수 케이스

**칭호 (영구 보유)**:
- **이미 획득한 칭호는 절대 박탈 안 됨** (유배 사유 제외)
- 경계값 조정 시 "새 기준으로는 얻기 더 어려워진 칭호" 보유자는 그대로 영광 유지
- 희소성이 오히려 높아짐 → 긍정적 부작용

**마패 (실시간 티어)**:
- Creator Score 변동에 따라 자동 조정
- 경계값 하향 시: 추가 유저 승격 → 축하 알림
- 경계값 상향 시: **grandfatheredMapae** 필드로 기존 티어 보장

```typescript
// users/{uid}
interface UserData {
  mapaeTier?: string;  // 현재 티어 (자동 계산)
  grandfatheredMapae?: string;  // 🆕 기득권 티어 (상향 조정 시만)
  grandfatheredMapaeAt?: FirestoreTimestamp;
}
```

## 📝 결론

글러브의 경계값 조정은 **감이 아닌 지표**로 결정합니다. 

**지금(Phase A)은 조정하지 않는 것이 정답입니다.** 유저가 너무 적어 분포 통계 자체가 의미 없습니다.

**Phase B(베타 종료 시점)에 데이터 기반으로 전면 재조정**합니다. 이때는 "리셋" 성격이므로 기득권 보장 없이 진행하되, 충분한 예고와 "베타 참여 칭호" 같은 상징적 보상으로 유저 경험을 관리합니다.

**Phase C(정식 출시 이후)부터는 A+B+E 전략 가동**합니다. 상향 조정 시 grandfatheredLevel로 기존 유저를 보호하고, 하향 조정은 즉시 적용해 유저 이익을 즉각 반영합니다.

Step 1에서는 **상수 파일 분리와 조정 근거 문서화**만 완료하고, 실제 조정 인프라(기득권 보장 필드, 일일 통계 수집, 대시보드)는 Step 2 이후에 구축합니다.
